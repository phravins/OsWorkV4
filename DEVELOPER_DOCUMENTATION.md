# OS Works Developer Documentation

Welcome to the OS Works codebase! This documentation breaks down exactly what every file and folder does within your Phoenix application. Phoenix uses a very standardized architecture, primarily separating the pure "business logic" (`lib/osworks`) from the "web presentation" (`lib/osworks_web`).

## Root Project Files

These files configure the entire project, its dependencies, and how it runs.

*   `docker-compose.yml`: Tells Docker how to spin up your application network alongside your PostgreSQL database container (`db`). This handles ports, environment variables, and the startup command so you don't have to map them manually.
*   `mix.exs`: The central configuration file for Elixir. It defines the application version, specifies the Elixir compilation paths, lists all third-party dependencies (like `phoenix`, `ecto`, `postgrex`), and defines custom mix aliases (like `mix setup`).
*   `mix.lock`: Automatically generated by `mix deps.get`. It locks down the exact versions of every dependency so that your application runs identically on any computer.
*   `.formatter.exs`: Configures the `mix format` tool. Tells Elixir what files should be automatically formatted to ensure clean, standardized code syntax across the team.

## Configuration Directory (`config/`)

This directory determines how the application behaves securely depending on the environment you are running it in (Development, Testing, or Production).

*   `config.exs`: The base configuration file. Settings defined here apply globally to all environments unless overridden specifically below. It defines your core application module, the main database connection parameters, the mailer (Swoosh), and the frontend build tools (esbuild and tailwind).
*   `dev.exs`: Configuration specifically for running on your computer locally (`MIX_ENV=dev`). Here we enable hot-code reloading (so the browser magically refreshes when you save an HTML file), disable expensive HTTPS requirements, expose verbose debug logging, and point the application to the local developer database (`host.docker.internal` via Docker). 
*   `test.exs`: Configuration for running the automated unit tests (`MIX_ENV=test`). It uses a specific test database (so testing doesn't scramble your dev data), disables the HTTP server entirely to save time, and disables email sending.
*   `runtime.exs`: Executed right as the production application boots up. It reads real-world secure Environment Variables (like `SECRET_KEY_BASE` and `DATABASE_URL`) that you don't want hardcoded into the source files.

## The Core Application: `lib/osworks`

This directory represents the "Brain" of your project. It knows absolutely nothing about HTTP requests, browsers, or HTML. It handles data processing, talking to the Database, and sending Emails.

*   `osworks.ex`: The entry point for context modules. Often holds global domain configurations or shared helper functions for your backend logic.
*   `osworks/application.ex`: The supervisor! When you start the application, this file runs first. It wakes up your Database Connection (Repo), starts your Web Server (Endpoint), and starts the Mailer. If any of them crash, it restarts them for you.
*   `osworks/repo.ex`: The Ecto adapter. This file is your bridge to the PostgreSQL database. Every time you save a user, search a table, or update a record, it passes through this module.
*   `osworks/mailer.ex`: The email adapter handling password resets, confirmations, and general outbound emails (using Swoosh).

### The Accounts Context (`lib/osworks/accounts/`)
This is the domain specifically dealing with user identity and security (generated securely via `phx.gen.auth`).
*   `accounts.ex`: The public interface for accounts. It contains functions like `get_user!/1`, `register_user/1`, or `authenticate_user/2`. If the web side needs anything from the accounts, it calls this file.
*   `user.ex`: The Ecto Schema defining how the "users" table looks in PostgreSQL (email, encrypted password, tokens). It also contains the exact logic for hashing passwords securely with Bcrypt.
*   `user_notifier.ex`: A module explicitly for building and sending the actual notification emails (like "Confirm your Email").
*   `user_token.ex`: The Ecto Schema handling session tokens, reset password tokens, and "remember me" functionality.
*   `scope.ex`: Often used by Ecto/authorization tooling to secure which database queries user accounts are allowed to run (preventing user A from editing user B).

## The Web Layer: `lib/osworks_web`

This directory handles everything the user actually sees: URLs, HTTP Requests, HTML, CSS, Javascript, and WebSockets.

*   `osworks_web.ex`: A macro file that defines shortcuts for your web modules. Every controller and live view injects this file so they automatically inherit HTML rendering functions and routing logic.
*   `endpoint.ex`: The front door of your web application. Before a request hits the Router, this file handles it: It parses incoming JSON, checks the session cookie, serves static files from `priv/static`, and configures the WebSocket.
*   `router.ex`: The traffic cop. It looks at the incoming URL (e.g., `GET /login`) and points it to the correct controller or LiveView (`UserController.login`). This file also groups routes into security "pipelines" (like ensuring the user is logged in before they see the dashboard).
*   `user_auth.ex`: High-level security functions utilized by the router (such as `require_authenticated_user` or `redirect_if_user_is_authenticated`).
*   `telemetry.ex` & `gettext.ex`: Handles internal metric tracking (how fast pages load) and translating the application text into different languages (localization).

### Controllers, HTML & Layouts (`lib/osworks_web/`)

*   **`controllers/page_controller.ex`**: Takes the traffic routed to the home page, does any necessary calculations, and then calls the HTML template. 
*   **`controllers/page_html/home.html.heex`**: The actual HTML markup of your root index homepage where your beautiful CSS design sits!
*   **`components/layouts/root.html.heex`**: The outer skeleton of the entire application. It contains the `<head>`, imports the CSS/JS files, and renders the generic `<header>` navigation bar you see on every page.
*   **`components/core_components.ex`**: Very important! Reusable user interface building blocks (like `<.button>`, `<.modal>`, `<.input>`). Any time you see a custom tag starting with a dot, it comes from this file.

### LiveViews (`lib/osworks_web/live/`)
Phoenix LiveView completely removes the need for building thick Javascript endpoints.

*   **`user_live/login.ex` & `user_live/registration.ex`**: These are real-time, stateful server-side rendered pages. If a user types an invalid email in the registration form, these files validate the input in real-time over WebSockets without reloading the webpage.

## Assets & Privilege Directories

*   **`assets/`**: Contains raw, uncompiled frontend assets: `app.css` (Tailwind inputs), `app.js` (Websocket logic). When you boot the application, esbuild and tailwind process these files.
*   **`priv/`**: Think of this as the compiled, secure output folder.
    *   `priv/static/`: Where the compiled `app.css`, `app.js`, and site logo images are copied to. These are the files actually served to the browsers.
    *   `priv/repo/migrations/`: Database schema version control. Every file here is a script (e.g., "Create the users table") that the application executes on PostgreSQL to ensure the database matches what the code expects.
